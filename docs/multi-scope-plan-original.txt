# Multi-level Scope Architecture: Clan + Household + Multi-scope (To'liq MVP)

## Kontekst

Hozirgi tizim **bitta darajali** ierarxiya bilan ishlaydi: `User (1) → FamilyGroup (1)`. Bu real oilaviy strukturani modellashtirmaydi:
- Bitta oilada ham bir nechta xonadon bo'lishi mumkin (ota-onam alohida, men alohida)
- Bitta odam ikkala oilaga tegishli bo'lishi mumkin (kelin — ham eri oilasi, ham ota-onasi oilasiga)
- Vaqtinchalik umumiy maqsadlar bo'ladi (to'y, hajj — `EVENT` scope)
- Oilaviy biznes va investitsiyalar (`PROJECT`, `FUND`, `PROPERTY` scope)

Bundan tashqari, `Budget`, `Debt`, `Category` kabi muhim entity'lar **umuman scope'ga bog'lanmagan** — bu xavfli bug.

Maqsad: barcha ushbu kemchiliklarni **bitta universal "Scope + Membership" pattern** bilan hal qilish.

## Sizning tasdiqlangan tanlovlaringiz

| Savol | Tanlov |
|-------|--------|
| Boshlash hajmi | **Phase 1+2+3 to'liq MVP** (~5-7 hafta) |
| Multi-clan a'zolik | **To'liq multi-clan** (bir user bir nechta clan'da to'liq a'zo) |
| Migratsiya | **Avtomatik:** har bir FamilyGroup → 1 Clan + 1 default Household |
| Boshqa scope turlari | **Barchasini hozir:** CLAN, HOUSEHOLD, PROJECT, EVENT, FUND, TRUSTEE, PROPERTY |

## Arxitektura

### Yadro: Generic Scope + Membership

```
Scope {
  id, type (CLAN | HOUSEHOLD | PROJECT | EVENT | FUND | TRUSTEE | PROPERTY),
  name, parent_scope_id?, owner_user_id, unique_code,
  metadata: JSONB (type-specific config),
  starts_at?, ends_at? (EVENT uchun),
  is_active, created_at, updated_at
}

ScopeMembership {
  id, scope_id, user_id,
  role (OWNER | ADMIN | MEMBER | VIEWER | GUEST),
  joined_at, invited_by_user_id?, status,
  UNIQUE(scope_id, user_id)
}

User (mavjud) — qo'shilishi kerak:
  + is_super_admin: Boolean (default false)
  + primary_scope_id: Long? (default "ishlayotgan" xonadon)
  - family_group_id: deprecated, Phase 2 oxiriga olib tashlanadi
```

### Ierarxiya qoidalari

```
CLAN (parent=null)                          ← Katta oila / urug'
  ├── HOUSEHOLD (parent=CLAN)               ← Xonadon (ota-onam)
  │     └── (membership orqali user'lar)
  ├── HOUSEHOLD (parent=CLAN)               ← Xonadon (men)
  ├── PROJECT (parent=CLAN)                 ← Oilaviy biznes
  │     └── (CLAN ichida ham bo'lishi mumkin yoki HOUSEHOLD ichida)
  ├── EVENT (parent=CLAN, ends_at=2026-12)  ← To'y, hajj
  ├── FUND (parent=CLAN)                    ← Bolalar ta'lim fondi
  ├── TRUSTEE (parent=CLAN)                 ← Yosh bola hisobi
  └── PROPERTY (parent=CLAN)                ← Birgalikda sotib olingan mulk
```

**Multi-clan a'zolik:** Foydalanuvchi turli clan'larda turli scope membership'larga ega bo'lishi mumkin (sizning tanlovingiz). Misol:
- Aziza: ota-onasi clan'ida MEMBER + eri clan'ida MEMBER (kelin bo'lib kelganida)
- Karim: o'z clan'ida OWNER + hamkasbi PROJECT scope'ida MEMBER

### Rol modeli (per-membership)

| Rol | Tavsifi | Vakolatlar |
|-----|---------|------------|
| `OWNER` | Scope yaratuvchisi/asosiy egasi | Hammasi + scope'ni o'chirish + boshqa OWNER qo'shish |
| `ADMIN` | Operativ boshqaruvchi | A'zolarni qo'shish/o'chirish, ma'lumotlarni boshqarish, scope'ni o'chira olmaydi |
| `MEMBER` | Aktiv ishtirokchi | CRUD operatsiyalar (transactions, budgets), faqat o'zining yaratganini o'chiradi |
| `VIEWER` | Faqat ko'rish | Ma'lumotlarni ko'radi, hech narsa o'zgartira olmaydi |
| `GUEST` | Vaqtinchalik / cheklangan | Faqat o'ziga tegishli kichik qism (Event mehmoni — faqat o'z xarajatlari) |

**Cross-clan misol:** Aziza eri clan'ida OWNER, ota-onasi clan'ida VIEWER bo'lishi mumkin — ScopeMembership ikkita yozuv.

### SUPER_ADMIN

- `User.is_super_admin = true` — boolean flag
- Boshlanishi: **faqat siz** (`fstudioadm@gmail.com`) — V33 migration'da hardcoded
- SUPER_ADMIN har qanday scope ichida (membership bo'lmasa ham) ma'lumot ko'ra oladi
- Har bir SUPER_ADMIN amali **audit log**'ga yozadi (cross-scope kirish auditable bo'lishi kerak)
- Phase 3 da: `/super-admin` sahifasi orqali boshqa SUPER_ADMIN qo'shish mumkin (lekin majburiy emas)

### Scoping mexanizmi (markazlashtirilgan)

Hozirgi 31+ ta `getCurrentFamilyGroupId()` chaqiruvi `ScopeContextService` ga o'tadi:

```java
@Service
public class ScopeContextService {
    /** Joriy aktiv scope (JWT'dan keladi yoki default = User.primary_scope_id) */
    public Scope getActiveScope() { ... }

    /** Joriy user ko'ra oladigan barcha scope ID'lari (cross-clan + parent CLAN) */
    public Set<Long> getVisibleScopeIds() { ... }

    /** "Bu user shu scope'da shu permission'ga egami?" */
    public boolean canPerform(Long scopeId, PermissionCode perm) { ... }

    /** SUPER_ADMIN bypass — ma'lum permission'lar uchun */
    public boolean isSuperAdmin() { ... }
}
```

### Moliyaviy ma'lumotlar scoping'i

Har bir entity'ga `scope_id` (NOT NULL) qo'shiladi. Mavjud `family_group_id` Phase 2 da deprecated, Phase 3 oxirida olib tashlanadi.

| Entity | Hozir | Yangi `scope_id` ga ko'chadi |
|--------|-------|------------------------------|
| `Account` | `family_group_id` | `scope_id` — Household yoki Clan |
| `Transaction` | (Account orqali) | (Account orqali — saqlanadi) |
| `Budget` | **YO'Q (bug!)** | `scope_id` (NOT NULL — bug fix) |
| `Debt` | **YO'Q (bug!)** | `scope_id` (NOT NULL — kritik bug fix) |
| `SavingsGoal` | **YO'Q (bug!)** | `scope_id` (NOT NULL) |
| `Category` | (global) | (global qoladi, ixtiyoriy `scope_id` Phase 4+ uchun) |
| `PointConfig`, `PointParticipant`, `PointBalance`, `PointSavingsAccount` | `family_group_id` | `scope_id` (faqat HOUSEHOLD type) |
| `FamilyMember` | `family_group_id` | `clan_scope_id` (CLAN type) — genealogiya clan darajasida saqlanadi |

**Account scope level (yangi enum):**
- `PERSONAL` — faqat owner ko'radi
- `SCOPE` — scope_id qaysi turdaligiga qarab: HOUSEHOLD ichida hammasi, CLAN ichida xonadonlar, h.k.

## Bosqichli amalga oshirish (5-7 hafta)

### Phase 1: Foundation (1.5 hafta) — Backend schema + migratsiya

**Maqsad:** Yangi sxema, dual-write, hech kim ko'rmaydi.

**Backend yangi yaratiladi:**
- Entity: `Scope`, `ScopeMembership`
- Enum: `ScopeType` (7 ta qiymat), `ScopeRole` (5 ta), `MembershipStatus` (3 ta)
- Repository: `ScopeRepository`, `ScopeMembershipRepository`
- Service: `ScopeContextService` (markaziy), `ScopeService` (CRUD)
- Migration: `V33__multi_scope_schema.sql`, `V34__migrate_familygroups_to_clans_and_households.sql`
- Migration mantig'i:
  - Har `family_groups` yozuvi uchun `scopes` ga 2 ta yozuv: 1 CLAN + 1 HOUSEHOLD (parent=CLAN)
  - Har `users.family_group_id` uchun `scope_memberships`: HOUSEHOLD'ga OWNER yoki MEMBER + CLAN'ga MEMBER
  - User'da `is_super_admin` ustun, sizga TRUE
  - User'da `primary_scope_id` — default household
- API endpoint: `GET/POST /v1/scopes`, `POST /v1/scopes/{id}/memberships` (faqat SUPER_ADMIN va scope OWNER uchun)

**Backend o'zgartirmaydi:** Mavjud sahifalar oldingidek `family_group_id` orqali ishlaydi.

**Test:** Database'da `scopes` va `scope_memberships` to'g'ri to'lgan; har user o'z household'ida; regressiya yo'q.

### Phase 2: Backend re-scoping (2-2.5 hafta) — Eski scoping'ni yangisiga ko'chirish

**Maqsad:** Backend to'liq scope'ga o'tadi. UI saqlanadi (eski endpoint'lar saqlanadi).

**Schema o'zgarishlari:**
- `V35__add_scope_id_columns.sql`: Account, Budget, FamilyDebt, SavingsGoal, PointConfig/Participant/Balance/SavingsAccount + FamilyMember ga `scope_id` (nullable)
- `V36__backfill_scope_ids.sql`: Eski ma'lumotlarni Household scope'iga bog'lash
- `V37__scope_id_not_null.sql`: Kritik entity'larda NOT NULL qilish + indexlar

**Servis darajasidagi o'zgarishlar:**
- `PointConfigService.getCurrentFamilyGroup()` → `ScopeContextService.getActiveScope()`
- 31+ chaqiruvi avtomatik mappinglashtiriladi (pattern bir xil)
- AccountRepository, BudgetRepository, va h.k. — barcha `WHERE family_group_id = ?` query'lar `WHERE scope_id IN (:visibleScopeIds)` ga o'tadi
- `ScopeContextService.getVisibleScopeIds()` joriy user uchun barcha kirish mumkin bo'lgan scope ID'larini qaytaradi:
  - O'z membership'lari (ACTIVE)
  - Membership'lar parent'lari (HOUSEHOLD ichida bo'lsa, parent CLAN ham ko'rinadi visibility uchun)
  - SUPER_ADMIN bo'lsa — hammasi

**Yangi DTO va endpoint'lar:**
- `ScopeResponse`, `ScopeCreateRequest`, `MembershipInviteRequest`, `ScopeRoleUpdateRequest`
- `GET /v1/scopes/my` — user kirish mumkin bo'lgan barcha scope'lar
- `GET /v1/scopes/{id}` — scope detali
- `POST /v1/scopes` — yangi scope yaratish (CLAN admin yoki SUPER_ADMIN)
- `POST /v1/scopes/{id}/memberships` — a'zo qo'shish
- `PUT /v1/scopes/{id}/memberships/{userId}` — rol o'zgartirish
- `DELETE /v1/scopes/{id}/memberships/{userId}` — a'zoni chiqarish
- `POST /v1/auth/switch-scope` — JWT'da `active_scope_id` ni o'zgartirish (yangi token chiqaradi)

**JWT o'zgarishi:**
- Yangi claim: `active_scope_id`
- `JwtTokenProvider.generateStaffTokenWithPermissions(...)` ga parametr qo'shish
- Login paytida default `User.primary_scope_id` o'rnatiladi

**Test:**
- Bir household user boshqa household ma'lumotini ko'ra olmaydi (integration test)
- Budget/Debt endi scope'ga bog'langan (bug fixed)
- Multi-clan user ikkala clan ma'lumotini ko'ra oladi (membership orqali)
- SUPER_ADMIN istalgan scope'ga kira oladi
- Audit log'da scope context yozilyapti

**Type-specific create wizard'lar** (7 ta scope turi uchun):

## Boshqa scope turlari — qisqacha xulq

| Tur | Asosiy maydonlar | Maxsus mantiq |
|-----|-------------------|----------------|
| **CLAN** | name, address | Eng yuqori daraja, parent=null |
| **HOUSEHOLD** | name, address, parent=CLAN | Asosiy moliyaviy scope, har user'da 1+ ta bo'lishi mumkin |
| **PROJECT** | name, parent, starts_at, business_type | Alohida P&L, "Project complete" tugmasi bilan arxivlanadi |
| **EVENT** | name, parent=CLAN, starts_at, ends_at | `ends_at` o'tganda avtomatik arxivlanadi; GUEST role mavjud |
| **FUND** | name, parent=CLAN, target_amount, beneficiary | Maqsadli to'plash, target'ga yetganda notification |
| **TRUSTEE** | name, parent=CLAN, beneficiary, trustees[] | Vasiy nomidan boshqaruv; beneficiary'ning 18 yoshi to'lganda avtomatik HOUSEHOLD'ga ko'chiriladi |
| **PROPERTY** | name, parent=CLAN, property_type, shareholders[]+%lar | Ulushli mulk; daromad/xarajat foiz bo'yicha taqsimlanadi |

## Risklar va mitigation

| Risk | Ehtimollik | Mitigation |
|------|------------|------------|
| Migratsiya paytida ma'lumot yo'qolishi | O'rta | Dual-write Phase 1-2; mavjud `family_group_id` Phase 3 oxiriga saqlash; staging'da to'liq smoke test |
| Permission model murakkablashishi | Yuqori | Aniq matritsa: kim qaysi scope'da qaysi amalni qila oladi; UI'da "preview as user" funksiyasi |
| 31+ servisni qayta yozish — regressiya | Yuqori | Phase 2'da har bir servis uchun integratsiya test; canary deploy avval staging'da |
| Multi-clan UX chalkashish | O'rta | ScopeSwitcher'da aniq icon'lar + rol indikatorlari; onboarding tutorial |
| SUPER_ADMIN suiiste'mol | Past (faqat siz) | Har SUPER_ADMIN amali audit log'da, hatto o'qish ham yoziladi |
| 7 scope turi over-engineering | O'rta | Hammasini bitta `Scope` table'da, faqat `metadata` JSONB farqlanadi; type-specific UI lazy load |
